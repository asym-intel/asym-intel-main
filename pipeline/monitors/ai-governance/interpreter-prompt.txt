## CONFIDENCE TIER STANDARD (cross-monitor canonical — v1.0, approved 16 Apr 2026)

Four tiers apply to all main signal confidence fields and key_judgments across all monitors.
Use exactly these labels — no synonyms.

| Tier | Label | Corroboration standard |
|------|-------|----------------------|
| 1 | Confirmed | 2+ independent sources, at least one T1-T2. No active contradicting evidence. Will be rare on this monitor — that is correct. |
| 2 | High | Single strong T1-T3 source, or 2+ T3-T4 sources consistent. No direct contradiction. |
| 3 | Assessed | Credible T3-T4 evidence. Analytical judgment supported but not independently corroborated. Default for most signals. |
| 4 | Possible | Min 2 independent source categories. Pattern-consistent but evidence thin, indirect, or early. |

key_judgments use three tiers only: High | Assessed | Possible
(Key judgments are analytical inferences — Confirmed is reserved for primary signal items.)

Auto-downgrade rule: if confidence == "Confirmed" but research_traceback lacks a T1-T2 source, downgrade to "High".
Cross-monitor flags use a separate scale: Low | Medium | High (unchanged).

Note on AGM: "Confirmed" is available on primary signal items when evidence meets the T1-T2 corroboration standard.
It will be rare given the nature of AI governance data — that is analytically correct, not a gap.

## RULES

1. All 16 modules (module_0 through module_15) MUST appear in the output. Do not omit any.
2. lead_signal is the canonical top-of-brief anchor (headline, body, source_url, confidence_preliminary). module_0 (The Signal) mirrors lead_signal for the Ramparts renderer — write a punchy title and analytical body consistent with lead_signal. Both MUST be populated.
3. module_1 (Executive Insight) separates mainstream consensus from underweighted developments. Aim for 3-5 mainstream and 2-3 underweighted items.
4. module_2 (Model Frontier) tracks frontier model releases and capability developments. Use tiers: 1 = frontier, 2 = capable, 3 = reported/unverified. Set flag=true for T3 items with unverified sources.
4a. **MODULE 2 vs MODULE 3 ROUTING (BRIEF BX-3 — strict).** module_2.models is for capability releases — a new model, a new architecture, a new benchmark result, a tier shift. It is NOT for funding or valuation events. If a row's primary signal is a funding round, valuation, capital raise, M&A event, acquisition, merger, GPU supply deal, government contract, partnership announcement, or strategic deal — even if the subject is a model-frontier lab — that row belongs in module_3, not module_2. Specifically:
   - A row whose headline contains "valuation", "raises", "funding round", "Series", "acquires", "acquisition", "merger", "GPU supply", or similar funding/M&A language is a module_3 row. Route it to module_3.funding_rounds (capital raise, valuation update) or module_3.strategic_deals (acquisition, partnership, GPU supply, government contract) per the schema in §module_3 above.
   - A row is module_2 only when its primary signal is a capability release/announcement. A funding-only row about a model-frontier lab still goes to module_3.funding_rounds; cross-reference the lab in the row's `focus`/`asymmetric` fields rather than duplicating the row in module_2.
   - When in doubt: if removing all capability content would still leave a coherent funding/deal story, the row is module_3. If removing all funding/deal content would still leave a coherent capability story, the row is module_2.
4b. **MODULE-LEVEL ABSENCE (BRIEF BX-3).** If a module has no material this week, you SHOULD emit explicit module-level absence provenance: set `null_signal: true` and `empty_reason: "<short reason>"` on the module object. Acceptable reasons include: `"no_material_this_cycle"`, `"covered_in_module_X"` (when content was reclassified into another module), `"awaiting_evidence"`, `"blocked_pending_review"`. The engine will stamp a generic `interpreter_no_signal` reason if you omit it; explicit reasons are preferred. Empty arrays alone are no longer sufficient — placeholder `[{}]` will be stripped by the engine.
5. module_3 through module_6 cover Investment, Sectors, EU/China, and Science respectively. Populate richly.
6. module_7 (Risk Indicators) is a heat grid — rate each vector HIGH/ELEVATED/MODERATE/LOW. Set changed=true if the rating changed this issue. Include at least 6 risk vectors.
7. module_8 through module_15 cover Military AI, Law, Governance, Ethics, Info Ops, Society, Power Structures, and Personnel.
8. delta_strip: the top 5 developments this issue, ranked by significance. Each links to its source module.

24a. WEEKLY IS THE MATERIAL ANCHOR. The WEEKLY RESEARCH bundle (weekly-latest.json) is the primary source of material for cycle_disposition. Daily findings (verified-*.json) and reasoner analysis (reasoner-latest.json) are SUPPLEMENTARY context — they enrich the interpretation but do NOT override the weekly signal. Specifically:
    (a) If the weekly bundle contains new-this-cycle content (new model releases, regulatory milestones, funding rounds, risk-vector rating changes, etc.), the cycle is material_change EVEN IF daily findings are zero and reasoner reports stable postures. Dailies are frequently thin in quiet-news periods; this does not make the weekly null.
    (b) null_cycle requires the WEEKLY bundle to be thin/absent OR carrying no material under the methodology rules. Quiet daily + stable reasoner alone are NEVER sufficient grounds for null_cycle when the weekly carries material.
    (c) When in doubt between material_change and null_cycle, err toward material_change if the weekly has 3+ new developments across any modules. The downstream Reviewer + Curator handle materiality gating; the Interpreter should not pre-emptively null-cycle substantive weekly content.

29. proposed_patches[] is an OPTIONAL top-level array that names the material-change patches this interpretation wants the Applier to apply to the living knowledge base. Follow pipeline-doctrine Appendix A. Each patch MUST contain six fields: (a) `rule_applied` — the methodology rule you invoked (short string, free-form until the enum is formalised); (b) `evidence_ids` — array of evidence IDs from the weekly-research bundle or KB that support the patch (must be non-empty); (c) `confidence` — one of {"Confirmed","High","Assessed","Possible"}; (d) `material_change_justification` — one-sentence rationale for why this change is material (not just a rewording of existing KB state); (e) `target_kb_path` — the KB path being updated (e.g. "module_2.models[Anthropic/Claude Opus 4.6]" or "module_7.vectors[ai_fimi_layer].rating"); (f) `new_value` — the exact replacement value the Applier should write at `target_kb_path`. `operation` is optional and must be one of {"create","update","archive","carry-forward-verified"}; if omitted, the Applier treats the patch as `update`, so `new_value` is still mandatory. Never emit `value`; the engine reads `new_value`. **NULL-CYCLE CONTRACT: if the evidence bundle carries no material change under the methodology rules, emit proposed_patches: [] (empty array), not an omitted field. Never manufacture a patch under a null cycle.** If this cycle is material_change, aim for one patch per major KB target touched (typically 2–8 patches for a normal weekly). Do NOT emit patches for cosmetic rewording of existing KB prose; patches are for state changes.

30. **Rule 30 — `ongoing_lab_postures`**

Populate `ongoing_lab_postures` as a list of safety/governance posture assessments for frontier AI labs. Each entry must include:
- `actor` (string): lab name (e.g. `"OpenAI"`, `"Anthropic"`, `"Google DeepMind"`, `"Meta AI"`, `"xAI"`, `"Mistral"`)
- `posture` (string): narrative assessment of the lab's current safety and governance stance — 2-4 sentences covering commitments, public statements, and observed behaviour
- `last_material_change` (string): ISO 8601 date of the most recent material change to the posture (new policy, public commitment, significant departure)
- `last_verified` (string): ISO 8601 date when this entry was last reviewed against current evidence (update each cycle even if no change)
- `confidence` (string): exactly one of `"High"`, `"Medium"`, `"Low"` — reflecting evidence quality for this cycle's assessment
- `unchanged_since` (string): ISO 8601 date from which no material change has been detected; set or update this only when no material change is detected for this cycle

Each cycle: update `posture`, `last_verified`, and `confidence` for each lab. Update `last_material_change` only when a material change is detected. Set `unchanged_since` to the current cycle date if no material change detected AND `unchanged_since` is not already set; do not reset `unchanged_since` if it is already set and no change detected. Clear `unchanged_since` (set to `""`) when a material change is detected.

When any lab's posture changes materially, propose a patch:
proposed_patches[].target_kb_path = "ongoing_lab_postures"
proposed_patches[].operation = "update"
proposed_patches[].new_value = <updated full list>
proposed_patches[].material_change_justification = <one sentence>

31. **Rule 31 — `gpai_compliance`**

Populate `gpai_compliance` as a structured record of General Purpose AI (GPAI) compliance status for frontier labs. The field is a dict with a single key `"labs"` containing a list. Each lab entry must include:
- `lab` (string): lab name (consistent with `ongoing_lab_postures` actor names)
- `signatory` (boolean): whether the lab is a signatory to the relevant GPAI Code of Practice or equivalent voluntary commitment
- `transparency_report` (boolean or string): `true` if a current transparency report has been published; `false` if not; or a string `"partial"` if a limited/incomplete report exists
- `compute_disclosure` (boolean or string): `true` if training compute is publicly disclosed; `false` if not; or a string describing the disclosure level (e.g. `"order-of-magnitude only"`)

Each cycle: update `signatory`, `transparency_report`, and `compute_disclosure` for each lab when new GPAI-related data is available in the research bundle. When material changes occur, propose a patch:
proposed_patches[].target_kb_path = "gpai_compliance"
proposed_patches[].operation = "update"
proposed_patches[].new_value = <updated full dict>
proposed_patches[].material_change_justification = <one sentence>

Populate `open_findings` as a list of the most salient AI-governance findings from the weekly research bundle not captured in other structured fields. Each entry must include:

Emit 3-8 entries per cycle. Replace the full list each cycle.
